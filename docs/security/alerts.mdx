## What is an alert?

An alert is created every time a span's attributes match one of ClaudeSec's [security rules](/security/rules). Each alert records:

| Field | Description |
|-------|-------------|
| `id` | Auto-incrementing integer |
| `ts` | ISO 8601 timestamp when the alert was created |
| `ruleLabel` | Name of the rule that fired (e.g., "AWS access key detected") |
| `severity` | `low`, `medium`, or `high` |
| `spanId` | ID of the span that triggered the alert |
| `traceId` | Session (trace) the span belongs to |
| `harness` | Which AI agent produced the span |
| `spanName` | Name of the span operation |
| `matchedText` | Up to 100 characters of the text that matched the regex |

## Viewing alerts in the dashboard

### Alerts tab

Click the **Alerts** tab in the ClaudeSec dashboard to see a live-updating table of all alerts, newest first. Each row shows the timestamp, rule label, severity badge, harness name, and matched text.

### Severity filter

Use the severity dropdown to narrow the list to `high`, `medium`, or `low` alerts only.

### Span context

Click any alert row to jump to the corresponding span in the graph view, where you can inspect all span attributes in the right panel.

## Desktop notifications

ClaudeSec uses the browser [Notifications API](https://developer.mozilla.org/en-US/docs/Web/API/Notifications_API) to fire desktop alerts when a high-severity threat is detected.

### Enabling notifications

1. The first time ClaudeSec detects a high-severity span, the browser will prompt you for notification permission.
2. Click **Allow**.
3. Future high-severity alerts will show as system notifications even when the ClaudeSec tab is in the background or minimized.

<Note>
  Notification permission is per-browser and per-origin. If you decline the prompt, you can re-enable it via your browser's site settings for `http://localhost:3000`.
</Note>

### Notification content

Each desktop notification shows:
- **Title**: the rule label (e.g., "HIGH: AWS access key detected")
- **Body**: the harness name and the first 100 characters of the matched text
- **Icon**: the ClaudeSec favicon

## Exporting alerts

### JSON export

Download a JSON file containing all current alerts:

**Via the dashboard**: Click **Export Alerts** in the Alerts tab toolbar.

**Via curl**:

```bash
curl -o alerts.json http://localhost:3000/api/alerts/export
```

The exported file has the following shape:

```json
{
  "exportedAt": "2024-01-15T14:23:00.000Z",
  "alerts": [
    {
      "id": 42,
      "ts": "2024-01-15T14:22:58.123Z",
      "ruleLabel": "AWS access key detected",
      "severity": "high",
      "spanId": "abc123def456",
      "traceId": "trace789",
      "harness": "claude-code",
      "spanName": "tool_use",
      "matchedText": "AKIAIOSFODNN7EXAMPLE"
    }
  ]
}
```

## Filtering alerts via the API

Retrieve alerts with optional filters:

```bash
# All alerts (up to 200, newest first)
curl http://localhost:3000/api/alerts

# Only high-severity alerts
curl "http://localhost:3000/api/alerts?severity=high"

# Limit to 50 alerts
curl "http://localhost:3000/api/alerts?limit=50"

# Combined
curl "http://localhost:3000/api/alerts?severity=medium&limit=100"
```

Response shape:

```json
{
  "alerts": [...],
  "total": 17
}
```

The `total` field reflects the total count matching the filter (not just the page returned).

## Clearing alerts

Clearing alerts removes all alert records from the database. Spans are not affected — the graph and sessions remain intact.

**Via the dashboard**: Click **Clear All Alerts** in the Alerts tab.

**Via curl**:

```bash
curl -X DELETE http://localhost:3000/api/alerts
```

<Warning>
  Alert deletion is permanent and cannot be undone. Export your alerts before clearing if you need to retain them for audit purposes.
</Warning>

## Alert lifecycle

```
Span arrives at POST /v1/traces
         ↓
Rule engine evaluates span attributes
         ↓
     Match found?
    /           \
  YES            NO
   ↓              ↓
Alert row         Span stored
created in        with severity='none'
alerts table
   ↓
Socket.io 'alerts-update' broadcast
   ↓
Browser receives event, refreshes Alerts tab
   ↓
High severity? → Desktop notification fired
```

## Integrating alerts into CI/CD

You can poll the alerts API from your own scripts to fail builds or send notifications when threats are detected during automated agent sessions:

```bash
#!/bin/bash
HIGH_COUNT=$(curl -s "http://localhost:3000/api/alerts?severity=high" | jq '.total')
if [ "$HIGH_COUNT" -gt "0" ]; then
  echo "FAIL: $HIGH_COUNT high-severity alerts detected"
  curl -s "http://localhost:3000/api/alerts/export" -o "security-report-$(date +%s).json"
  exit 1
fi
echo "PASS: No high-severity alerts"
```
